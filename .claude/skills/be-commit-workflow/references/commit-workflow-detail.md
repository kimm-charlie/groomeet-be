# Commit Workflow Detail

## 커밋 메시지 형식

```
#{브랜치번호} {type}: {설명}
```

- 브랜치 번호: 현재 브랜치명에서 추출
  - 예: `feature/#532-chat-api` → `#532`
  - 예: `refactor/#565-string-validation` → `#565`
- type (4가지만 사용):
  - `feature`: 새로운 기능
  - `refactor`: 리팩토링
  - `chore`: 기타 변경 (설정, 의존성 등)
  - `test`: 테스트 추가/수정
- 설명: 한국어, 간결하게, 마침표 없이

### 예시
```
#532 feature: 채팅방 조회 API 추가
#532 chore: 채팅방 DTO 네이밍 수정
#565 refactor: String 타입 검증 방법 변경
#540 test: 배너 조회 RestDocs 테스트 추가
```

## 커밋 시점

- 빌드 성공(`./gradlew clean build`) 후 커밋한다
- task 단위 작업 완료 및 빌드 성공 시 자동으로 커밋한다

## 커밋 전략

- 관련 변경사항을 의미 있는 단위로 묶는다
- 리뷰어가 이해할 수 있는 논리적 단위로 분리한다
- plan 기반 작업 시 task별로 커밋한다

## 체크리스트

- [ ] 빌드 성공 확인
- [ ] 의미 있는 커밋 메시지 작성
- [ ] 올바른 형식 (`#{번호} {type}: {설명}`) 준수
- [ ] 리뷰어가 변경 의도를 이해할 수 있는지 확인
